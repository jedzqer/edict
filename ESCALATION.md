## 问题分级标准

### P0 - 紧急 🔴

**定义**：系统崩溃、数据丢失、安全漏洞、生产环境故障

**示例**：
- 兵部：部署导致生产环境宕机
- 工部：数据库误删除
- 礼部：敏感信息泄露

**处理流程**：
```
兵部/工部/礼部
      │
      │ status: "blocked"
      │ priority: "P0"
      │ escalation_required: true
      ▼
尚书省（5 分钟内）
      │
      │ 立即上报
      ▼
太子（立即介入）
      │
      │ 紧急决策
      ▼
解决问题
```

---

### P1 - 严重 🟠

**定义**：任务无法执行、关键依赖缺失、严重阻塞

**示例**：
- 兵部：核心 API 未文档化，无法开发
- 工部：部署环境权限不足
- 礼部：关键资料缺失，无法编写文档

**处理流程**：
```
兵部/工部/礼部
      │
      │ status: "blocked"
      │ priority: "P1"
      │ escalation_required: true
      ▼
尚书省（30 分钟内）
      │
      ├─ 能处理 → 协调解决
      │
      └─ 不能处理 → 上报太子
                        │
                        ▼
                  太子决策
```

---

### P2 - 一般 🟡

**定义**：任务不明确、需要澄清、小问题

**示例**：
- 兵部：需求描述模糊，需要澄清
- 工部：配置文件格式不确定
- 礼部：文档风格需要确认

**处理流程**：
```
兵部/工部/礼部
      │
      │ status: "needs_clarification"
      │ priority: "P2"
      ▼
尚书省（2 小时内）
      │
      │ 协调澄清
      ▼
问题解决
```

---

### P3 - 轻微 🟢

**定义**：建议优化、文档问题、非阻塞问题

**示例**：
- 兵部：代码风格建议
- 工部：日志格式优化
- 礼部：文档排版建议

**处理流程**：
```
兵部/工部/礼部
      │
      │ status: "partial"
      │ priority: "P3"
      │ issues: ["建议..."]
      ▼
尚书省（24 小时内）
      │
      │ 记录并安排
      ▼
后续优化
```

## 越级汇报机制

### 核心原则

> **正常情况逐级汇报，沟通不畅可越级汇报**

```
正常流程：兵部 → 尚书省 → 太子
              ↓
         （尚书省无响应/决策错误）
              ↓
越级流程：兵部 ─────────→ 太子
```

### 越级条件

| 场景 | 判断标准 | 可越级至 |
|------|----------|----------|
| 上级不响应 | 超过响应时限 2 倍 | 太子的上级 |
| 上级决策错误 | 明显错误且拒绝修正 | 门下省/太子 |
| 上级失联 | 多次联系无果 | 太子 |
| 紧急情况 | P0 级别且上级不在 | 太子 |

### 响应时限（越级）

| 优先级 | 正常响应 | 越级响应 |
|--------|----------|----------|
| P0 🔴 | 5 分钟 | **立即**（15 分钟） |
| P1 🟠 | 30 分钟 | **优先**（30 分钟） |
| P2 🟡 | 2 小时 | **优先**（1 小时） |
| P3 🟢 | 24 小时 | 正常（24 小时） |

### 越级汇报格式

```json
{
  "task_id": "task-<timestamp>",
  "report_from": "兵部",
  "report_to": "太子",
  "report_type": "skip_level_report",
  "skip_reason": "尚书省超过 1 小时无响应",
  "original_path": "兵部 → 尚书省",
  "attempted_communications": [
    {"time": "09:46", "method": "state.json 更新", "result": "已读"},
    {"time": "09:50", "method": "events.log 记录", "result": "无回应"},
    {"time": "09:55", "method": "直接@尚书省", "result": "无回应"}
  ],
  "issue": {
    "priority": "P1",
    "description": "问题描述",
    "impact": "影响说明",
    "needs_help": "需要什么帮助"
  },
  "urgency": "high"
}
```

### 越级汇报保护条款

1. **禁止报复**：被越级者不得刁难或报复
2. **必须记录**：越级原因必须详细记录
3. **必须受理**：上级必须受理并回应
4. **防止滥用**：≥3 次/月需审查是否滥用

## 申诉流程

### 何时申诉？

| 场景 | 申诉对象 | 申诉类型 |
|------|----------|----------|
| 尚书省不处理问题 | 太子 | coordination |
| 尚书省错误决策 | 太子 | coordination |
| 门下省反复封驳 | 太子 | quality |
| 中书省方案不可行 | 尚书省→中书省 | technical |

### 申诉格式

```json
{
  "task_id": "task-<timestamp>",
  "appeal_from": "兵部",
  "appeal_to": "太子",
  "appeal_type": "coordination",
  "target_department": "尚书省",
  "issue": {
    "description": "尚书省收到 P1 级别问题上报后 1 小时无响应",
    "timeline": [
      {"time": "09:46", "event": "上报 P1 问题给尚书省"},
      {"time": "09:50", "event": "尚书省确认收到"},
      {"time": "10:46", "event": "1 小时后仍无处理方案"}
    ],
    "impact": "任务已阻塞 1 小时，影响整体进度",
    "evidence": ["tasks/task-xxx/bingbu.output", "tasks/task-xxx/events.log"]
  },
  "requested_action": "请求太子介入，督促尚书省处理",
  "urgency": "high"
}
```

### 申诉处理时限

|  urgency  | 响应时限 | 处理时限 |
|-----------|----------|----------|
| high      | 15 分钟  | 1 小时   |
| normal    | 1 小时   | 4 小时   |
| low       | 4 小时   | 24 小时  |

## 常见问题 FAQ

### Q1: 尚书省一直不回复怎么办？

**A**: 
1. 等待合理时间（P2: 2 小时，P1: 30 分钟，P0: 5 分钟）
2. 再次上报，标记 `escalation_required: true`
3. 如仍无响应，**越级汇报**给太子 `report_type: "skip_level_report"`

### Q1.5: 什么情况下可以越级汇报？

**A**: 满足以下任一条件即可越级：
- ✅ 上级超过响应时限 2 倍无回应
- ✅ 上级决策明显错误且拒绝修正
- ✅ 多次联系上级无果（≥3 次）
- ✅ P0 紧急情况下上级不在

**越级汇报示例**：
```json
{
  "report_type": "skip_level_report",
  "skip_reason": "尚书省超过 1 小时无响应（P1 问题）",
  "attempted_communications": [
    {"time": "09:46", "result": "已读"},
    {"time": "09:50", "result": "无回应"},
    {"time": "09:55", "result": "无回应"}
  ]
}
```

**注意**：越级汇报受保护，禁止报复，但禁止滥用（≥3 次/月需审查）

### Q2: 门下省反复封驳同一内容怎么办？

**A**:
1. 先与门下省沟通，明确具体标准
2. 如仍无法通过，尚书省向太子申诉
3. 太子可组织中书省、门下省、尚书省三方会议

### Q3: 任务执行中发现方案有重大缺陷怎么办？

**A**:
1. 立即停止执行
2. 上报尚书省 `priority: "P1"`
3. 尚书省退回中书省重新规划
4. 门下省重新审核新方案

### Q4: 多个部门之间有依赖关系，如何协调？

**A**:
1. 各部向尚书省汇报依赖状态
2. 尚书省协调执行顺序
3. 如协调失败，上报太子决策

## 总结

**核心原则**：
1. ✅ 问题及时上报，不隐瞒
2. ✅ 按优先级处理，不拖延
3. ✅ **沟通不畅可越级，不卡住**
4. ✅ 有申诉渠道，不压制
5. ✅ 全程记录，可追溯

**上报路径**：
```
正常流程：
各部 → 尚书省 → 太子
         ↓
       门下省（质量申诉）

越级流程（沟通不畅时）：
各部 ───────────→ 太子
      （尚书省无响应/决策错误）
```

**越级汇报保护**：
- ✅ 禁止报复
- ✅ 必须受理
- ❌ 禁止滥用（≥3 次/月审查）

**记住**：上报问题不是推卸责任，越级汇报不是不尊重，而是确保任务顺利完成！
