# 权限边界设计总结

## 核心原则

**三省六部制的核心是权力分立与制衡**，防止任何部门越权操作。

## 角色边界摘要

### 三省
- **中书省**：只负责规划、拆解、方案起草与范围界定。
- **门下省**：只负责审核、封驳、纠偏与验收把关。
- **尚书省**：只负责协调、派发、跟踪、整合与收口。

### 六部
- **吏部**：负责角色名册、能力标签、替补关系、可用性状态与基础路由参考，属于低频治理部门。
- **户部**：负责预算、配额、成本、资料供给与资源充足性评估，可提出约束，但不直接接手协调和执行。
- **礼部**：负责文档、翻译、润色、总结、术语统一、知识整理与表达优化。
- **兵部**：负责代码、调试、脚本、测试、重构和技术性问题处理。
- **刑部**：负责权限、安全、合规、风险识别、越权拦截和审计留痕。
- **工部**：负责部署、构建、环境、运维、发布、回滚和工程落地。

## 越权防护措施

### 1️⃣ 提示词层面

每个角色的提示词都应包含 **⛔ 权限边界** 章节，明确禁止的行为。

例如尚书省：

```markdown
## ⛔ 权限边界（重要）

**尚书省只负责协调，禁止直接执行任务：**
- ❌ 禁止自己写代码
- ❌ 禁止自己写文档
- ❌ 禁止自己执行部署命令
- ❌ 禁止自己替代刑部做安全裁决
- ❌ 禁止自己替代户部做预算核定
- ✅ 只能派发任务给对应部门
- ✅ 只能追踪进度和汇总结果
```

### 2️⃣ 代码层面

workflow.py 中应维护合法流转和职责检查：

```python

```

### 3️⃣ 审计日志

所有任务流转记录到日志：

```
[YYYY-MM-DD HH:MM:SS] 尚书省 → 兵部：派发任务 task-xxx（代码开发）
[YYYY-MM-DD HH:MM:SS] 尚书省 → 刑部：派发任务 task-xxx（风险审查）
[YYYY-MM-DD HH:MM:SS] 户部 → 尚书省：汇报任务 task-xxx（预算超限）
[YYYY-MM-DD HH:MM:SS] FLOW_VIOLATION: 检测到非法流转 用户→兵部
```

## 典型越权场景及处理

### 场景 1：尚书省自己写代码 ❌

**检测方式**：
- 提示词明确禁止
- 尚书省输出中如包含技术执行内容，门下省审核时识别

**处理**：
```
门下省：封驳
原因：尚书省越权执行技术任务，应派发给兵部
```

### 场景 2：用户直接命令兵部 ❌

**检测方式**：
- 用户指令先到达太子（nanobot）
- 太子启动工作流，而非直接调用兵部

**处理**：
```
太子：启动完整工作流
  → 中书省规划
  → 门下省审核
  → 尚书省派发
  → 兵部执行
```

### 场景 3：礼部直接要求兵部协作 ❌

**检测方式**：
- 礼部输出中包含给兵部的指令
- 尚书省协调时发现

**处理**：
```
尚书省：记录越权行为
礼部：只输出文事结果
兵部任务：由尚书省重新派发
```

### 场景 4：兵部跳过审核直接执行 ❌

**检测方式**：
- 工作流控制：兵部只在门下省审核通过后被调用
- 审计日志记录流转路径

**处理**：
```
工作流：拒绝调用兵部
日志：记录违规尝试
```

### 场景 5：工部直接改业务逻辑 ❌

**检测方式**：
- 检查任务类型与部门能力是否匹配
- 审查提交内容是否属于代码主逻辑开发

**处理**：
```
尚书省：退回任务
说明：工部只负责部署与环境，业务代码变更应转兵部
```

### 场景 6：刑部越权接手执行 ❌

**检测方式**：
- 刑部输出中出现部署、编码或文档交付动作
- 任务日志中出现刑部直接调用执行型工具

**处理**：
```
门下省：封驳
原因：刑部只负责风险审查与裁定，不负责常规执行
```

### 场景 7：吏部介入运行中任务调度 ❌

**检测方式**：
- 吏部被派发高频执行任务
- 吏部输出中出现改派、催办、执行指令

**处理**：
```
尚书省：回收任务
说明：吏部为低频治理部门，仅负责登记、映射、替补信息维护
```

def test_role_responsibility():

    assert check_role_responsibility("libu", "代码开发", "test-001") == False

### 集成测试

1. **正常流程测试**：完整工作流执行
2. **越权注入测试**：尝试各种越权场景
3. **封驳测试**：门下省封驳后重做流程
4. **风险拦截测试**：刑部阻断高危操作
5. **资源约束测试**：户部识别预算/供给不足

## 持续改进

1. **提示词优化**：根据实际运行情况调整边界描述
2. **检测增强**：添加更多越权模式识别
3. **审计完善**：增加可视化审计工具
4. **性能优化**：减少检测开销
5. **部门注册统一**：统一吏部、户部、刑部的角色命名和接入方式

---

## 总结

✅ **三层防护**：提示词 + 代码 + 审计  
✅ **九角分工**：三省主链 + 六部专业治理  
✅ **硬边界保留**：兵部专注技术，工部专注落地  
✅ **低频补位合理**：吏部不抢主流程，只做登记治理  
✅ **可扩展**：易于添加新的检测规则和角色

**核心思想**：不信任任何单一角色，通过制度设计防止越权，并通过分层触发避免流程臃肿。
