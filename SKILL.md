# edict-dev · 三省六部制 AI 协作系统 (开发分支)

🏛️ 基于中国古代三省六部制设计的多 Agent 协作框架。

> **⚠️ 开发分支**: 本分支用于测试新架构（LangGraph 重构），生产环境请使用 `edict` 主分支。

## 架构

```
用户 → 太子（nanobot/你） → 判断复杂度
    ├─ 简单请求 → 直接回复
    └─ 复杂任务 → edict 工作流
                  ↓
            📜 中书省（规划起草）
                  ↓
            🔍 门下省（审核封驳）
                  ↓
            📮 尚书省（协调派发）
                  ↓
    ┌─────────┬─────────┬─────────┬─────────┬─────────┬─────────┐
    ↓         ↓         ↓         ↓         ↓         ↓
  🏷️ 吏部   💰 户部   🎨 礼部   ⚔️ 兵部   ⚖️ 刑部   🔧 工部
  编制治理   资源供给   文事表达   技术实现   风险审查   工程落地
```

## 角色职责

| 角色 | 定位 | 执行引擎 | 典型场景 |
|------|------|----------|----------|
| 📜 **中书省** | 需求理解、任务拆解、方案起草 | spawn | 复杂任务规划、范围界定、执行路径设计 |
| 🔍 **门下省** | 审核封驳、质量把关、纠偏验收 | spawn | 方案审查、结果验收、遗漏检查 |
| 📮 **尚书省** | 总协调、任务派发、结果整合 | spawn | 多部门协作、跨部门依赖处理、统一交付 |
| 🏷️ **吏部** | 角色名册、能力标签、替补登记 | spawn | 角色注册、职责映射、可用性登记（低频） |
| 💰 **户部** | 预算配额、资源供给、成本评估 | spawn | 成本控制、资料充足性检查、资源不足判断 |
| 🎨 **礼部** | 文档、写作、翻译、润色、知识整理 | spawn | 文档、说明、翻译、总结、表达优化 |
| ⚔️ **兵部** | 代码、技术实现、调试修复 | **opencode** | 编程、架构、脚本、测试、修复 |
| ⚖️ **刑部** | 权限审查、安全合规、风险拦截 | spawn | 危险操作审查、敏感数据、越权检测、审计 |
| 🔧 **工部** | 部署、构建、环境、运维落地 | **opencode** | 发布、环境配置、CI/CD、服务维护 |

## 部门边界说明

### 三省：流程主干
- **中书省**：负责想清楚、拆清楚、写清楚方案。
- **门下省**：负责审清楚、挑问题、定标准。
- **尚书省**：负责派清楚、盯进度、收结果。

### 六部：专业执行与治理
- **吏部**：低频治理部门，主要维护部门名册、能力标签、替补关系和基础路由参考，不参与常规高频执行。
- **户部**：负责预算、配额、成本、资料供给和资源是否充足的判断，给出更经济的执行建议。
- **礼部**：负责一切文事类任务，包括文档、翻译、润色、总结、术语统一、知识整理和对外表达。
- **兵部**：只负责技术实现本身，包括写代码、改代码、调试、测试、脚本和技术性问题处理。
- **刑部**：负责风险与边界，包括权限、安全、合规、越权拦截、敏感操作审查和审计留痕。
- **工部**：只负责工程落地，包括部署、构建、环境、运维、发布、回滚和依赖处理。

> **边界原则**：兵部做“程序内容本身”，工部做“运行环境与发布落地”；礼部做“表达加工”，门下省做“审核把关”；刑部做“风险裁断”，户部做“资源供给”。

### 作为太子（nanobot）启动工作流

当用户请求复杂任务时：

```
1. 判断任务复杂度
2. 调用 workflow.py (传统) 或 langgraph_workflow.py (LangGraph) 启动 edict
3. 中书省规划，门下省审核，尚书省协调
4. 按需触发六部执行或治理
5. 汇总并交付最终结果
```

### 触发方式

**手动触发**（用户明确请求）：
- "用 edict 处理这个任务"
- "启动三省六部工作流"
- "请用多 Agent 协作完成"

**自动触发**（太子判断）：
- 涉及多步骤的复杂任务
- 需要多领域专业知识
- 需要质量审核的重要任务
- 同时涉及代码、文档、部署、风险或资源约束

## 推荐触发机制

### 固定主链
复杂任务默认经过：
1. **中书省**：规划
2. **门下省**：审核
3. **尚书省**：协调

### 高频执行部门
按任务内容触发：
- **礼部**：文档、翻译、说明、总结、表达加工
- **兵部**：代码、脚本、调试、技术分析
- **工部**：部署、环境、构建、运维

### 条件治理部门
按需触发：
- **刑部**：涉及危险操作、权限、敏感信息、生产环境、高风险变更
- **户部**：涉及预算、配额、多模型成本、资料不足、资源不足
- **吏部**：角色变更、能力登记、替补查询时低频触发

## 目录结构

⚔️ **兵部任务由 OpenCode 执行**，利用其专业的代码理解和编辑能力。  
🔧 **工部任务也可由 OpenCode 执行**，负责环境、部署和工程落地操作。

### 前置要求

```bash
# 确保 opencode 已安装
opencode --version

# 确保 PATH 包含 /usr/sbin（opencode 需要 sysctl）
export PATH="/usr/sbin:/usr/bin:/sbin:/bin:$PATH"
```

### 调用方式

```python
from edict.workflow import run_opencode_task

# 简单任务
output = run_opencode_task(
    task_id="task-xxx",
    task_description="添加用户认证功能"
)

# 指定项目目录
output = run_opencode_task(
    task_id="task-xxx",
    task_description="优化数据库查询",
    project_dir="/path/to/project"
)
```

### OpenCode 优势

- ✅ 完整的项目上下文理解
- ✅ 多文件编辑能力
- ✅ 会话保持（继续之前的工作）
- ✅ 支持 plan/build 双模式
- ✅ JSON 输出便于集成

## 封驳机制

门下省有权封驳不合格的方案或结果：

```json
{
  "decision": "封驳",
  "issues": ["步骤缺失", "部门分配错误"],
  "suggestions": "补充部署步骤，高风险操作需经刑部审查，资源超限任务需户部评估"
}
```

封驳后任务会打回相应阶段重做。

## 最佳实践

### 何时使用 edict

✅ 适合：
- 多步骤复杂任务
- 需要专业分工的任务
- 需要质量审核的重要任务
- 跨领域协作任务
- 涉及代码 + 文档 + 部署 + 风险控制的复合任务

❌ 不适合：
- 简单查询（直接回复）
- 单一步骤任务
- 无需拆解的轻量请求
- 紧急且无需协作的即时回答

### 太子判断准则

```
简单请求 → 直接回复
- 事实查询
- 简单计算
- 单轮对话

复杂任务 → edict 工作流
- 项目创建
- 系统设计
- 多文件代码
- 文档 + 代码组合
- 部署发布流程
- 高风险或高成本任务
```

## 示例

### 示例 1：创建项目

**用户**：帮我创建一个 Python Flask API 项目，包含用户认证和数据库

**太子判断**：复杂任务 → 启动 edict

**工作流**：
1. 中书省：拆解为数据库设计 → API 开发 → 认证实现 → 测试 → 部署准备
2. 门下省：审核方案合理性与验收标准
3. 尚书省：派发兵部（代码）、工部（部署），必要时户部评估成本
4. 兵部：编写 Flask 代码
5. 工部：配置 Docker 部署
6. 门下省：最终验收
7. 交付用户

### 示例 2：文档 + 代码

**用户**：为我的项目写技术文档并优化代码性能

**工作流**：
- 礼部：编写技术文档
- 兵部：优化代码性能
- 尚书省：协调两者依赖
- 门下省：验收文档准确性和代码改进

### 示例 3：高风险发布

**用户**：把这个服务上线到生产环境并替换现有配置

**工作流**：
- 中书省：制定发布方案
- 门下省：审核步骤与回滚方案
- 户部：评估资源条件与执行成本
- 刑部：审查权限、风险与敏感操作
- 工部：执行部署变更
- 尚书省：协调与收口

## 扩展

### 添加新角色

1. 在 `roles/` 创建 `{role}.md`
2. 定义职责和输出格式
3. 在 `workflow.py` 中注册
4. 在权限矩阵和协议中补充合法流转

### 自定义工作流

修改 `workflow.py` 中的 `run_workflow()` 函数，调整执行顺序、触发条件或添加新阶段。

## 注意事项

- 子代理通过 `spawn` 调用，异步执行
- 兵部/工部通过 `opencode` 执行专业工程任务
- 任务状态通过文件系统持久化
- 封驳可能导致多次重试
- 刑部和户部应按条件触发，不建议无差别串行加入每次任务
- 吏部属于低频治理部门，建议只做登记、映射与替补管理
- 复杂任务耗时较长，建议告知用户预期时间
