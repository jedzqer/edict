## 任务流转规则

### ✅ 合法流转

| 来源 | 目标 | 允许 | 说明 |
|------|------|------|------|
| 用户 | 太子 | ✅ | 用户发起任务 |
| 太子 | 中书省 | ✅ | 太子启动工作流 |
| 中书省 | 门下省 | ✅ | 提交方案审核 |
| 门下省 | 尚书省 | ✅ | 批准方案后派发 |
| 尚书省 | 吏部/户部/礼部/兵部/刑部/工部 | ✅ | 派发执行或治理任务 |
| 吏部/户部/礼部/兵部/刑部/工部 | 尚书省 | ✅ | 汇报执行结果 |
| 尚书省 | 门下省 | ✅ | 提交验收 |
| 门下省 | 太子/用户 | ✅ | 最终交付 |

> **注**：尚书省仍是唯一正式派发入口；六部均不得绕过尚书省直接互相派发任务。

### ❌ 非法流转（越权）

| 来源 | 目标 | 禁止 | 原因 |
|------|------|------|------|
| 尚书省 | 直接写代码/部署/写文档 | ❌ | 尚书省负责协调，不直接执行 |
| 兵部 | 直接接收用户指令 | ❌ | 绕过审核与派发 |
| 礼部 | 直接向兵部派发任务 | ❌ | 绕过尚书省 |
| 工部 | 直接向中书省汇报 | ❌ | 绕过尚书省 |
| 户部 | 直接否决已批准方案并改派任务 | ❌ | 户部只做资源判断，不做总协调 |
| 刑部 | 直接接管执行任务 | ❌ | 刑部负责审查，不负责常规执行 |
| 吏部 | 直接干预运行中任务 | ❌ | 吏部只做低频登记治理 |
| 任何部门 | 跳过门下省审核 | ❌ | 绕过质量检查 |

## 部门职责与通信边界

### 三省
- **中书省**：可起草方案，不可直接派发执行。
- **门下省**：可审核封驳，不可直接执行任务。
- **尚书省**：可协调派发，不可直接替代六部做专业执行。

### 六部
- **吏部**：维护角色名册、能力标签、替补关系与可用性状态；仅接受登记、映射、查询类任务。
- **户部**：负责预算、资源、配额、资料供给与成本评估；可提出资源约束意见，不直接改写技术方案。
- **礼部**：负责文档、翻译、润色、总结、术语统一、知识整理与对外表达。
- **兵部**：负责代码、脚本、调试、测试、技术实现；不负责部署上线。
- **刑部**：负责权限、安全、合规、风险审查与审计留痕；可阻断高风险操作，但不直接接手执行。
- **工部**：负责部署、构建、环境、运维、发布与回滚；不负责主要业务代码开发。

## 异常上报机制

### 问题分级

| 级别 | 类型 | 上报路径 | 响应时限 |
|------|------|----------|----------|
| **P0 - 紧急** | 系统崩溃、数据丢失、安全漏洞、重大越权 | 尚书省 → 太子（立即） | 5 分钟 |
| **P1 - 严重** | 任务无法执行、依赖缺失、严重阻塞、高风险变更冲突 | 尚书省 → 太子（优先） | 30 分钟 |
| **P2 - 一般** | 任务不明确、需要澄清、资源不足、小范围错误 | 尚书省处理 | 2 小时 |
| **P3 - 轻微** | 建议优化、文档问题、登记偏差 | 尚书省记录 | 24 小时 |

### 越级汇报机制

**原则**：正常情况下逐级汇报，**沟通不畅时可越级汇报**

| 场景 | 正常路径 | 越级条件 | 越级路径 |
|------|----------|----------|----------|
| 尚书省不响应 | 部 → 尚书省 | 超过响应时限 2 倍 | 部 → 太子 |
| 尚书省决策错误 | 部 → 尚书省 | 明显错误且拒绝修正 | 部 → 门下省 |
| 门下省反复封驳 | 尚书省 → 门下省 | 同一问题 ≥ 3 次 | 尚书省 → 太子 |
| 中书省方案不可行 | 尚书省 → 中书省 | 拒绝修改且无替代 | 尚书省 → 太子 |
| 刑部判定高风险但协调未处理 | 刑部 → 尚书省 | 仍要求继续执行 | 刑部 → 太子 |

**越级汇报响应时限**：
- P0/P1 越级：**立即响应**（15 分钟内）
- P2/P3 越级：**优先响应**（1 小时内）

## 越权处理

### 检测到越权时

1. **记录日志**：记录越权行为的详细信息
2. **拒绝执行**：不执行越权指令
3. **上报太子**：向太子（nanobot）报告
4. **流程回滚**：回到正确的流转路径
5. **必要时转刑部**：若涉及敏感数据、权限、生产环境或重大风险，由刑部出具风险结论

## 申诉机制

### 适用场景

| 场景 | 说明 | 申诉路径 |
|------|------|----------|
| 尚书省不处理问题 | 问题上报后无响应 | 六部 → 太子 |
| 尚书省错误决策 | 决策明显不合理 | 六部 → 门下省/太子 |
| 门下省反复封驳 | 同一问题多次打回 | 尚书省 → 太子 |
| 中书省方案不可行 | 技术/资源不可行 | 尚书省 → 中书省（要求重规划） |
| 刑部风险结论被无视 | 仍要求继续危险操作 | 刑部 → 太子 |

### 申诉处理原则

1. **不越级优先**：优先向直接上级申诉
2. **有证据**：申诉必须附带证据
3. **不报复**：禁止因申诉而刁难
4. **有回应**：申诉必须在 1 小时内回应
5. **风险优先**：刑部确认高风险时，未复核前不得继续执行

### 越权检测示例

```python
# 伪代码示例
def validate_dispatch(source, target, task):
    legal_flows = {
        ("尚书省", "吏部"): True,
        ("尚书省", "户部"): True,
        ("尚书省", "礼部"): True,
        ("尚书省", "兵部"): True,
        ("尚书省", "刑部"): True,
        ("尚书省", "工部"): True,
        ("用户", "兵部"): False,
        ("礼部", "兵部"): False,
        ("刑部", "工部"): False,
    }

    if (source, target) not in legal_flows:
        log_violation(source, target, task)
        reject_task(task)
        report_to_taizi(source, target, task)
```

## 审计日志

所有跨部门通信必须记录到审计日志：

```
[YYYY-MM-DD HH:MM:SS] 尚书省 → 兵部：派发任务 task-xxx（代码开发）
[YYYY-MM-DD HH:MM:SS] 尚书省 → 刑部：派发任务 task-xxx（高风险命令审查）
[YYYY-MM-DD HH:MM:SS] 刑部 → 尚书省：汇报任务 task-xxx（风险等级：高，建议暂停）
[YYYY-MM-DD HH:MM:SS] 户部 → 尚书省：汇报任务 task-xxx（预算超限，建议降级执行）
[YYYY-MM-DD HH:MM:SS] 审计：检测到非法流转 用户→兵部，已拒绝
```
